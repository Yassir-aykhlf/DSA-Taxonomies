# Heap & Priority Queue

> A heap keeps the smallest (or largest) element at the root, giving O(log n) insert/extract and O(1) peek — ideal when you repeatedly need the current extreme.

> **Use when**
> - You need the top-K (largest/smallest/most-frequent) elements without fully sorting.
> - You merge K sorted lists/streams and always want the next-smallest across all of them.
> - You track a running statistic on a stream — a median (two heaps) or a moving extreme.
> - You schedule tasks/events by priority, or run Dijkstra/Prim where the next node is the cheapest.
> - You greedily combine items by repeatedly pulling and re-inserting the current extreme.

> **Complexity**
> - Push / pop: **O(log n)**; peek: **O(1)**; heapify a list of n: **O(n)**.
> - Top-K over n items with a size-K heap: **O(n log k)** time, **O(k)** space.
> - K-way merge of N total elements across k lists: **O(N log k)** time.

## Related patterns

- **Greedy Algorithms** — heaps implement the "always take the current best" choice.
- **Graph Algorithms** — Dijkstra's and Prim's pull the cheapest frontier node from a heap.
- **Sliding Window** — some windowed extreme/median problems lean on heaps.
- **Two Pointers** — an alternative when the data is sorted and no heap is needed.
